# ROOTLINE 开发路线图与 AI 执行文档

## 1. 文档目的

这份文档用于交接给专门负责开发实现的 AI。它基于当前项目已有的两份设计文档：

- `docs/ROOTLINE_Game_Design_Document.md`
- `docs/ROOTLINE_Technical_Design_Document.md`

ROOTLINE 的核心不是传统地图探索，而是一个“洞穴拓扑驱动文明演化”的地下 CRPG。开发时应优先证明核心系统链路：

> 玩家改变洞穴连接关系 -> 资源、稳定性、派系关系、迁徙、冲突和历史随之改变。

所有功能都应服务于这个核心循环。不要过早扩大到完整 CRPG、美术资产、大规模内容或复杂战斗。

## 2. 会话轮数定义

本文中的“1 轮会话”指一次相对完整的开发闭环，通常包括：

1. 阅读相关上下文。
2. 设计当轮目标。
3. 实现代码或文档。
4. 运行必要验证。
5. 总结结果和下一步。

预估轮数不是工期承诺，而是用于拆解复杂度和控制节奏。

## 3. 总体阶段划分

建议将 ROOTLINE 分为 8 个 Phase 完成。

- 可玩 MVP：Phase 0 到 Phase 4，预计 18-28 轮会话。
- 完整原型 Demo：Phase 5 到 Phase 8，再追加 20-32 轮会话。
- 总体完整原型：预计 38-60 轮会话。

优先目标是先做出可玩 MVP，而不是一次性追求完整游戏。

## 4. Phase 0：项目基础与方向锁定

预计轮数：1-2 轮会话。

当前状态：基本已完成。

### 目标

- 初始化 Git 仓库。
- 建立项目文档。
- 明确游戏核心幻想。
- 锁定第一个可玩版本的范围。

### 已有产出

- GitHub 仓库：`git@github.com:onovich/Rootline.git`
- 游戏设计文档。
- 技术设计文档。
- 项目计划文档。

### 验收标准

- 仓库可 clone。
- 文档说明清楚游戏的设计方向、技术方向和 MVP 目标。

## 5. Phase 1：技术选型与工程骨架

预计轮数：2-4 轮会话。

### 目标

- 决定第一版技术栈和运行形态。
- 建立项目目录结构。
- 配置测试、格式化、lint 或基础验证命令。
- 提供一个可运行的最小空壳。

### 技术建议

优先选择能最快验证模拟系统的方案：

- 如果优先快速做可视化：TypeScript + Web + Canvas/DOM ASCII。
- 如果优先终端风格和低复杂度：Python 或 Rust + TUI。
- 如果未来明确要游戏化界面：TypeScript + Canvas/Pixi 或 Godot。

推荐第一选择：TypeScript + Web 前端。

原因：

- 容易做 ASCII/图形混合界面。
- 容易展示事件日志、节点详情、调试面板。
- 后续可以演进为 Electron 桌面版。

### 交付物

- 可运行项目骨架。
- 基础数据模型目录。
- 测试命令。
- 本地启动命令。
- README 中记录开发命令。

### 验收标准

- 新开发者或 AI 能通过 README 启动项目。
- 至少有一个空白或 debug 页面可以运行。
- 至少有一个基础测试可以通过。

## 6. Phase 2：洞穴图与 Tick 模拟核心

预计轮数：4-7 轮会话。

### 目标

- 实现 cave graph。
- 实现确定性的 tick engine。
- 实现资源流动。
- 实现稳定性变化和坍塌事件。
- 引入 seedable randomness，保证同 seed 同结果。

### 核心数据对象

- Node：洞穴节点。
- Edge：洞穴连接或隧道。
- Resource：食物、氧气、热量等资源。
- Stability：结构稳定性。
- Event：模拟事件。
- TickState：当前模拟状态。

### 最小规则

- 开放边允许资源流动。
- 阻塞边禁止通行和资源流动。
- 坍塌边需要事件记录。
- 节点稳定性过低时可能触发坍塌。
- 固定 seed 下模拟结果必须可复现。

### 交付物

- 手写小型洞穴图。
- Tick 引擎。
- 资源流动规则。
- 稳定性和坍塌规则。
- 模拟日志。
- 单元测试。

### 验收标准

- 连接或切断一条边会改变资源分布或生存结果。
- 固定 seed 多次运行得到同样事件序列。
- 测试覆盖图修改、资源流动、稳定性更新和坍塌。

## 7. Phase 3：ASCII 可玩循环

预计轮数：5-8 轮会话。

### 目标

- 做出第一版可视化和交互。
- 玩家可以探索、查看节点、连接、封闭、等待 tick。
- 加入 fog of war。
- 加入事件日志。
- 做出 10-20 分钟可玩的垂直切片。

### 核心操作

- Explore：探索未知节点。
- Inspect：查看节点、边、派系、资源和稳定性。
- Connect：打开或建立连接。
- Sever：封闭或切断连接。
- Wait：推进 tick。
- View Log：查看历史事件。

### 交付物

- ASCII 洞穴图视图。
- 节点详情面板。
- 边状态显示。
- 雾区/已知区域显示。
- 事件日志。
- 基础交互命令。

### 验收标准

- 玩家能完成“探索 -> 遇到派系 -> 改变连接 -> 观察后果”的循环。
- 屏幕上能看懂当前世界状态。
- 重大变化都有事件解释。

## 8. Phase 4：派系与信仰系统

预计轮数：5-9 轮会话。

### 目标

- 定义 belief schema。
- 实现至少 3 个派系。
- 让派系根据信仰决定行动倾向。
- 让拓扑变化影响信任、敌意、迁徙和禁忌违反。

### 首批派系建议

1. Echo Cult
   - 信仰：洞穴储存记忆。
   - 禁忌：破坏墙体。
   - 反应：反感过度挖掘或坍塌。

2. Excavator Union
   - 信仰：扩张等于生存。
   - 倾向：主动开路和挖掘。
   - 反应：支持连接新区域，讨厌封闭路线。

3. Strata Guild
   - 信仰：结构稳定性是神圣的。
   - 倾向：维护稳定，防止坍塌。
   - 反应：反对高风险挖掘，支持加固。

### 核心系统

- BeliefSystem：定义允许行为、偏好、禁忌和世界观。
- FactionPolicy：将信仰转化为行为权重。
- RelationsModel：记录信任、敌意、依赖、禁忌违反。
- FactionEvent：记录派系态度变化和行动原因。

### 交付物

- 信仰系统数据结构。
- 派系策略逻辑。
- 派系关系模型。
- 派系行动事件。
- 事件解释文本。

### 验收标准

- 同样的拓扑变化，不同派系会出现不同反应。
- 玩家能看懂派系为什么生气、迁移、结盟或开战。
- 派系行为不只是随机数，而是来自信仰和当前世界状态。

## 9. Phase 5：任务、叙事与历史层

预计轮数：4-7 轮会话。

### 目标

- 把 quest 设计成状态转化目标，而不是固定脚本。
- 记录 NPC、派系和洞穴拓扑的历史。
- 让事件日志逐步形成文明史。
- 增加目标型场景。

### 任务示例

- 连接两个孤立派系。
- 阻止危险信仰传播。
- 保证某个节点群存活一定 tick 数。
- 触发一次安全迁徙。
- 避免核心区域发生坍塌。

### 交付物

- Emergent quest system。
- History log。
- Scenario objectives。
- 结局或阶段性叙事摘要。

### 验收标准

- 每局结束能总结出一段“这个地下文明发生了什么”。
- 任务目标来自系统状态，而不是单纯脚本触发。
- 玩家行动能明显改变历史摘要。

## 10. Phase 6：战斗原型

预计轮数：5-8 轮会话。

### 目标

- 实现 squad / NPC combat。
- 用图拓扑决定 line of sight、掩体、瓶颈和包围。
- 让坍塌、封路和开路成为战术手段。
- 确保战斗服务于文明模拟，而不是吞掉项目重心。

### 交付物

- Squad model。
- NPC combat stats。
- Tick-resolved combat。
- Graph LOS。
- Combat event log。
- 战斗调试视图。

### 验收标准

- 战斗胜负受到连接关系、通道状态、节点稳定性影响。
- 玩家可以通过改变地形制造优势或灾难。
- 战斗结果会反过来影响派系关系和历史。

## 11. Phase 7：AI 生成与模拟验证

预计轮数：6-10 轮会话。

### 目标

- 生成洞穴图。
- 生成派系与信仰。
- 批量运行模拟。
- 自动筛掉无聊、不稳定或不可玩的 seed。
- 输出可玩场景和叙事摘要。

### 核心流程

1. AI 或程序生成 cave graph。
2. 生成 factions 和 beliefs。
3. 运行 N 次模拟。
4. 收集稳定性、冲突、迁徙、资源危机和事件密度。
5. 评分。
6. 输出可玩 seed。

### 交付物

- World generator。
- Faction/belief generator。
- Simulation validator。
- Seed scoring report。
- 可玩 seed 导出格式。

### 验收标准

- 能批量生成 20-100 个 seed 并自动评分。
- 能解释 seed 为什么可玩或不可玩。
- 至少能筛出数个适合试玩的场景。

## 12. Phase 8：存档、场景包装与 Demo 打磨

预计轮数：6-10 轮会话。

### 目标

- 实现完整 save/load。
- 定义 scenario package 格式。
- 打磨 UI/交互。
- 平衡首批 playable scenarios。
- 准备可分享 demo。

### 交付物

- Save system。
- Scenario package。
- Demo build。
- README 操作说明。
- 2-3 个稳定可玩的场景。

### 验收标准

- 玩家可以保存、读取并继续模拟。
- 存档后继续运行仍保持确定性。
- Demo 可以交给其他人试玩，不需要开发者现场解释。

## 13. 推荐执行顺序

优先完成 MVP：

1. Phase 1：技术选型与工程骨架。
2. Phase 2：洞穴图与 tick 模拟核心。
3. Phase 3：ASCII 可玩循环。
4. Phase 4：派系与信仰系统。

完成 MVP 后再进入：

5. Phase 5：任务、叙事与历史层。
6. Phase 6：战斗原型。
7. Phase 7：AI 生成与模拟验证。
8. Phase 8：存档、场景包装与 Demo 打磨。

## 14. 开发 AI 的执行原则

- 每一轮只完成一个明确目标。
- 优先实现可验证的系统，不要先写大量设定。
- 所有模拟逻辑必须尽量可测试、可复现、可解释。
- 不要过早引入复杂美术、复杂剧情或大量手写内容。
- 每个重大系统都应有调试输出或事件解释。
- 保持 simulation core 与 presentation layer 分离。
- 如果功能不能让“拓扑改变文明”更清晰，就暂缓。
