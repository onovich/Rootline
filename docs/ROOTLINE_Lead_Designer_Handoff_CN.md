# ROOTLINE 主策划交接文档

## 1. 交接目的

这份文档用于把此前由架构师临时兼任的主策划职责正式交接给主策划角色。主策划后续负责 ROOTLINE 的整体游戏体验、内容方向、策划标准和策划线推进；架构师继续负责技术边界、系统结构、数据契约和工程验收。

交接后的核心协作目标：

> 主策划决定游戏为什么值得玩，架构师确保这些设计能被稳定实现，执行策划把设计转成可落地内容，执行程序把内容和系统做成可验证软件。

## 3. 当前共同认知

ROOTLINE 是一个 simulation-first 的地下文明 CRPG。它的核心不是传统地图，而是可变洞穴图：

- 节点是洞穴。
- 边是通道。
- 连接、切断、坍塌、封闭和开掘会改变资源、稳定性、派系关系、迁徙、冲突和历史。

最重要的设计命题：

> 洞穴不是地理，而是拓扑。玩家改变拓扑，文明因此变化。

项目当前必须坚持以下共识：

- 拓扑是玩法核心。
- 模拟是事实来源。
- ASCII/debug 表现先于最终美术。
- 信仰系统必须影响派系行为。
- 任务是状态转化目标，不是固定脚本。
- 叙事来自 NPC、派系和洞穴拓扑的历史演化。
- AI 生成内容必须经过模拟验证。
- 角色入口可以克制，但游戏内容必须极其丰富。

## 4. 主策划职责边界

主策划负责：

- 定义核心体验和玩家幻想。
- 决定角色系统的方向和差异。
- 决定派系、信仰、禁忌、文化冲突和内容密度。
- 决定任务、事件、场景和历史叙事的体验标准。
- 定义什么叫好玩、可读、可重玩、有意义失败。
- 为执行策划制定内容生产标准。
- 为架构师和程序员提供可系统化的设计要求。

主策划不直接负责：

- 具体代码实现。
- 技术选型。
- 模块边界。
- 测试框架。
- 性能架构。
- 存档格式的最终工程实现。

主策划可以提出系统需求，但必须交由架构师确认是否符合现有技术方向。

## 5. 已确认的主策方向

### 5.1 角色系统方向

玩家不再只是单一观察者，而应有不同可选角色。

每个角色应至少包含：

- 职业。
- 属性。
- 背景。
- 出生地点。
- 初始已知地图。
- 初始派系关系。
- 特殊行动。
- 限制或风险。
- 个人目标。
- 初始剧情钩子。

首批建议角色：

- 根脉测绘员。
- 地层工程师。
- 放逐调停者。
- 破壁开拓者。

MVP 阶段不急着做完整成长树。优先让角色影响开局、信息、行动、派系关系和个人目标。

### 5.2 内容丰富度方向

ROOTLINE 应采用“小角色入口，大内容世界”：

- 首批角色：3-4 个。
- 首批派系：5-6 个。
- 长期派系：12 个以上。
- 首批区域生态：6-8 种。
- 首批事件模板：50-80 个。
- 长期事件模板：200 个以上。
- 首批任务目标类型：20-30 个。
- 首批重要 NPC 原型：30 个左右。
- 首批遗物和旧世界内容：30-50 个。
- 首批高密度场景：3 个。

丰富度必须来自可复用内容池，而不是一次性剧情堆砌。

## 6. 必须对齐的设计协议

### 6.1 可系统化协议

所有策划内容必须能被系统读取或实现。主策划不得只交付氛围设定。

每个关键设计条目应尽量包含：

- `id`。
- `name`。
- 类型或标签。
- 触发条件。
- 数值范围。
- 状态变化。
- 玩家可见反馈。
- 派系或 NPC 反应。
- 失败或副作用。
- 示例。
- 验收标准。

不满足这些字段的内容只能作为灵感，不得进入执行策划或程序实现阶段。

### 6.2 模拟优先协议

主策划提出的新内容必须回答：

- 它改变了哪个模拟状态？
- 它由什么状态触发？
- 它如何被玩家感知？
- 它如何影响未来 tick、派系关系、资源、稳定性或历史？

如果一个内容不改变状态，也不帮助玩家理解状态，则应暂缓。

### 6.3 拓扑核心协议

ROOTLINE 的主要玩法必须围绕洞穴连接关系展开。

主策划设计角色、任务、事件、派系和场景时，都应至少回答一个问题：

- 这个内容如何让连接、切断、封闭、坍塌、开路、可达性或隔离变得重要？

不能服务拓扑核心的内容可以存在，但不得成为 MVP 的主要工作。

### 6.4 角色差异协议

角色差异不能只靠数值加成。

每个角色至少需要在以下 5 个维度中体现 3 个：

- 不同出生点。
- 不同初始信息。
- 不同可用行动。
- 不同派系关系。
- 不同个人目标或结局评价。

### 6.5 事件与任务协议

事件必须是状态触发，任务必须是状态目标。

事件模板应包含：

- 触发条件。
- 参与对象。
- 结果效果。
- 玩家可见文本。
- 历史日志文本。
- 权重或冷却建议。

任务目标应包含：

- 初始状态。
- 目标状态。
- 成功条件。
- 失败条件。
- 可接受代价。
- 多解法路径。

### 6.6 内容命名协议

所有内容条目应使用稳定 ID。

建议格式：

- 角色：`role.rootline_surveyor`
- 派系：`faction.echo_cult`
- 区域：`region.echo_hall`
- 事件：`event.collapse_warning`
- 任务：`quest.connect_isolated_factions`
- NPC：`npc.echo_warden_01`
- 遗物：`relic.old_lift_core`

显示名可以中文或英文，但 ID 必须稳定、可引用、可搜索。

## 7. 流程依赖关系

### 7.1 主策划与架构师

主策划依赖架构师确认：

- 新设计是否符合 simulation-first 架构。
- 新字段是否应该进入核心数据模型。
- 新系统是否会破坏现有模块边界。
- 新内容是否需要新增技术阶段。
- 哪些设计可以先做内容表，哪些必须等系统支持。

架构师依赖主策划提供：

- 明确的体验目标。
- 可系统化字段。
- 内容优先级。
- MVP 与非 MVP 边界。
- 设计验收标准。

协议：

- 主策划每个 Phase 开始前输出本地计划文档。
- 架构师对计划中的数据结构、系统边界、风险进行审查。
- 未经过架构师确认的数据契约，不应交给程序员实现。

### 7.2 主策划与执行策划

主策划负责方向，执行策划负责产量和可实现细节。

主策划应交给执行策划：

- 内容主题。
- 内容池规模。
- 字段模板。
- 风格边界。
- 验收标准。
- 禁止事项。

执行策划应交回：

- 内容表。
- 示例条目。
- 触发条件。
- 数值范围。
- 玩家可见文本。
- 系统效果。
- QA 清单。

协议：

- 执行策划不得擅自改变核心体验方向。
- 主策划不得只给抽象概念，必须给可填表的模板。
- 每批内容至少经过一次“系统化检查”和“玩家感知检查”。

### 7.3 主策划与执行程序

主策划不直接给程序员模糊设定。

交给程序员前，策划内容必须先具备：

- 稳定 ID。
- 数据字段。
- 触发条件。
- 状态效果。
- 验收标准。
- 至少一个示例。

协议：

- 程序员只应实现已被主策划和架构师确认的内容契约。
- 如果程序员发现设计无法实现，应退回给架构师和主策划，不应自行改设计方向。
- 程序员完成后，架构师验收系统边界，主策划验收体验目标。

## 8. Skill 使用协议

当前项目已有两个通用 skill：

- `GoalNext`，调用名：`$goal-next`
- `DoNextGoal`，调用名：`$do-next-goal`

主策划使用要求：

1. 制定下一阶段计划时，使用 `$goal-next`。
2. 当前会话角色应声明为 `主策划`。
3. 所有计划必须输出到本地 `docs/` 文档。
4. 文档必须包含目标、非目标、角色路由、阶段轮次、自检标准、验收标准和下一 AI 交接 prompt。

执行策划使用要求：

1. 执行具体内容生产时，使用 `$do-next-goal`。
2. 当前会话角色应声明为 `执行策划`。
3. 每完成一轮并通过自检后，必须 commit 并 push 到远端。
4. 推送成功后才允许进入下一轮。

架构师使用要求：

1. 审查技术计划或系统边界时，使用 `架构师` 角色。
2. 必须检查模拟优先、数据契约、模块边界、测试策略。
3. 不通过时应给出阻塞原因和最小修复建议。

执行程序使用要求：

1. 实现代码时，使用 `执行程序` 角色。
2. 必须遵守架构师确认的数据契约。
3. 每轮通过测试和自检后推送远端。

## 9. 主策划每轮自检标准

主策划每轮产出必须检查：

### 目标检查

- 本轮目标是否完成。
- 是否偏离当前 Phase。
- 是否明确 MVP 与非 MVP。

### 体验检查

- 玩家是否能感知这个设计。
- 是否产生选择。
- 是否有代价、风险或后果。
- 是否服务“拓扑改变文明”。

### 系统化检查

- 是否有字段。
- 是否有触发条件。
- 是否有状态变化。
- 是否能交给执行策划或架构师继续推进。

### 内容丰富度检查

- 是否能扩展成内容池。
- 是否与角色、派系、事件、任务、区域产生交叉。
- 是否避免一次性剧情堆砌。

### 协议检查

- 是否符合已有 GDD / TDD / Project Plan。
- 是否符合 `docs/ROOTLINE_Design_Work_Roadmap_CN.md`。
- 是否需要架构师确认数据模型。
- 是否需要程序员提供技术能力支持。

## 12. 决策升级协议

以下情况必须升级给架构师和用户确认：

- 设计改变 MVP 核心循环。
- 新增大系统，例如完整对话树、装备成长、复杂经济系统。
- 改变玩家身份定义。
- 改变核心资源轴。
- 改变派系行为模型。
- 要求程序侧改变架构层级。
- 内容规模超出当前 Phase 预算。

以下情况主策划可自行决定：

- 内容命名和风格微调。
- 单个派系、事件、NPC、区域的文本表达。
- 不改变数据结构的内容扩写。
- 符合模板的示例条目补充。

## 13. 成功标准

主策划工作成功，不是因为设定写得多，而是因为：

- 程序员能实现。
- 执行策划能批量扩展。
- 玩家能感知差异。
- 系统能产生后果。
- 每局能形成历史。
- 内容丰富但不破坏架构。

ROOTLINE 的主策划应始终守住一句话：

> 让每一次连接或切断，都像是在改写一段地下文明史。
