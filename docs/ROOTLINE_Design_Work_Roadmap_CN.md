# ROOTLINE 策划工作路线图与执行标准

## 1. 文档目的

这份文档用于交接给主策划、执行策划或负责内容设计的 AI。它与技术侧开发路线并行推进，目标是让 ROOTLINE 不只是一个可运行的模拟器，而是一个内容密度足够高、可重玩、可产生历史的地下文明 CRPG。

策划线的核心原则：

> 小角色入口，大内容世界。

首批玩家角色可以克制在 3-4 个，但世界内容、派系、事件、任务、NPC、遗物、区域生态和历史生成必须形成可持续扩展的内容池。

所有策划内容都必须能被系统执行。不要只写氛围设定；每个设计条目都应尽量包含字段、触发条件、数值范围、玩家反馈、系统后果、示例和验收标准。

## 2. 与技术线的关系

程序线负责让系统成立：

- 洞穴图。
- tick 模拟。
- 派系状态。
- ASCII/交互层。
- 存档与生成验证。

策划线负责让系统值得玩：

- 玩家身份与角色差异。
- 高密度派系与世界内容。
- 可系统触发的事件。
- 状态目标式任务。
- NPC、遗物、旧世界碎片。
- 可量化的可玩性标准。

两条线并行，但通过文档和数据字段对齐。

## 3. 总体阶段与会话轮数

建议将策划工作拆成 10 个 Phase。

- 策划 MVP 内容：Phase 0 到 Phase 5，预计 18-28 轮会话。
- 高密度 Demo 内容：Phase 6 到 Phase 8，追加 17-27 轮会话。
- 长期内容圣经第一版：Phase 9 持续推进，总体预计 55-85 轮会话。

每一轮必须先完成本轮自检，确认通过后提交并推送远端，再进入下一轮。

## 4. Phase 0：策划规则与内容格式建立

预计轮数：1-2 轮会话。

### 目标

- 建立策划文档结构。
- 规定所有内容必须可系统化。
- 统一字段、命名、标签、数值范围和触发条件格式。

### 产出

- 策划总规范。
- 内容条目模板。
- 事件、派系、角色、任务、NPC、区域、遗物的字段标准。
- 策划内容命名规则。

### 验收标准

- 程序 AI 能根据字段实现数据结构。
- 策划 AI 后续可以持续扩写内容。
- 不存在只有氛围、不能实现的设定。
- 每类内容都有最小可用模板。

## 5. Phase 1：核心体验与玩家角色系统

预计轮数：3-5 轮会话。

### 目标

- 明确玩家不是单一观察者，而是多个可选角色。
- 每个角色拥有职业、属性、背景、出生点、初始关系、个人目标和信息差。
- 首批设计 4 个高差异角色，暂不扩展复杂成长树。

### 首批角色建议

1. 根脉测绘员
   - 擅长探索、地图、结构信息。
   - 初始已知区域较大。
   - 派系关系偏中立。

2. 地层工程师
   - 擅长稳定性、加固和坍塌预测。
   - 与 Strata Guild 初始关系较好。
   - 激进挖掘会带来更高声誉风险。

3. 放逐调停者
   - 擅长外交、信任修复和社会压力识别。
   - 出生在派系交界或争议区域。
   - 战斗和工程能力较弱。

4. 破壁开拓者
   - 擅长开路、爆破和高风险拓扑干预。
   - 与 Excavator Union 初始关系较好。
   - 更容易触发坍塌、禁忌和派系敌意。

### 产出

- `PlayerRole` 设计表。
- 属性系统。
- 起始地点设计。
- 角色背景与个人目标。
- 角色专属行动、限制和信息差。
- 角色与派系初始关系矩阵。

### 验收标准

- 每个角色不只是数值不同，而是玩法不同。
- 每个角色能改变开局、信息、行动和派系关系。
- 角色设计可以被程序转成数据结构。
- 同一场景下不同角色至少有 2 种明显不同解法。

## 6. Phase 2：派系内容圣经

预计轮数：5-8 轮会话。

### 目标

- 设计 8-12 个主派系。
- 每个派系包含信仰、禁忌、资源需求、外交逻辑、内部矛盾和代表 NPC。
- 首批实现 5-6 个，长期扩展到 12 个以上。

### 派系池建议

- Echo Cult。
- Excavator Union。
- Silent Order。
- Strata Guild。
- Mycelium Collective。
- Machine Remnants。
- Ash Breathers。
- Rootline Monks。
- Oxygen Cartel。
- The Unmapped。
- Broken Crown Colony。
- Deep Archive Keepers。

### 产出

- 派系总表。
- 信仰字段。
- 禁忌规则。
- 行动倾向。
- 初始关系矩阵。
- 派系事件模板。
- 派系代表 NPC 原型。

### 验收标准

- 同一事件下，不同派系反应明显不同。
- 每个派系都有系统行为，不只是背景设定。
- 派系之间存在可模拟的冲突、依赖、误解和交易。
- 每个派系至少有 3 个可触发事件钩子。

## 7. Phase 3：洞穴生态与区域内容

预计轮数：4-7 轮会话。

### 目标

- 设计地下世界的区域类型。
- 每种区域影响资源、稳定性、视野、行动风险和派系行为。
- 让地图生成和场景设计拥有足够素材。

### 区域池建议

- 根脉圣所。
- 干涸氧穴。
- 热泉矿层。
- 菌网腹地。
- 机械遗迹。
- 坍塌王座。
- 回声厅。
- 黑水裂谷。
- 盐骨墓层。
- 旧电梯井。
- 盲根森林。
- 深层档案库。

### 产出

- Biome / Region 设计表。
- 资源分布规则。
- 风险规则。
- 区域事件。
- 区域视觉与 ASCII 表现建议。
- 区域与派系偏好关系。

### 验收标准

- 区域不是皮肤，而会改变玩法。
- 每种区域至少影响 2-3 个系统。
- 程序可将区域规则转成模拟参数。
- 每种区域至少有 3 个事件或任务钩子。

## 8. Phase 4：资源、社会、空间系统策划

预计轮数：4-6 轮会话。

### 目标

- 完整定义 survival / social / spatial 三类系统。
- 让资源危机、社会崩溃、通路变化都能触发事件。
- 提供可测试数值区间。

### 系统轴

- 生存：食物、氧气、热量、水、医疗。
- 社会：信任、影响力、凝聚力、恐惧、异端压力。
- 空间：连通性、稳定性、可达性、封闭度、坍塌风险。

### 产出

- 资源系统表。
- 节点状态规则。
- 边状态规则。
- 危机触发条件。
- 数值区间建议。
- 资源短缺后果表。

### 验收标准

- 每个资源都有来源、消耗、流动和短缺后果。
- 每个社会指标都能影响派系或 NPC 行为。
- 每个空间指标都和拓扑变化有关。
- 规则能被单元测试或模拟验证。

## 9. Phase 5：事件模板库

预计轮数：6-10 轮会话。

### 目标

- 建立大规模事件池。
- 事件不是固定剧情，而是由状态触发。
- 首批设计 50-80 个事件模板，长期扩展到 200 个以上。

### 事件类型

- 资源危机。
- 坍塌灾害。
- 信仰传播。
- 派系冲突。
- 迁徙。
- 背叛。
- 遗物发现。
- 疫病。
- 谣言。
- 禁忌破坏。
- 旧机器苏醒。
- 地图认知错误。

### 产出

- 事件模板字段。
- 触发条件。
- 参与派系和 NPC 条件。
- 结果效果。
- 玩家可见文本。
- 历史日志文本。
- 事件权重和冷却建议。

### 验收标准

- 每个事件都有明确触发条件和后果。
- 事件能被模拟系统调用。
- 事件文本能解释为什么发生。
- 至少 30% 的事件能与角色、派系或区域产生组合变化。

## 10. Phase 6：任务与历史叙事系统

预计轮数：4-7 轮会话。

### 目标

- 把任务设计成状态转化目标。
- 每局能生成文明史。
- 玩家行动能改变结局摘要。

### 任务类型

- 连接两个孤立派系。
- 阻止某种信仰传播。
- 维持某节点群存活。
- 促成迁徙。
- 防止核心节点坍塌。
- 让敌对派系共存。
- 保护或摧毁旧世界遗物。

### 产出

- 任务目标库。
- 历史日志格式。
- 结局摘要规则。
- 成功、失败和代价评价标准。
- 多解法任务模板。

### 验收标准

- 任务不是 NPC 发一句话，而是系统状态目标。
- 同一任务可以有多种解法。
- 结局摘要能反映玩家实际行为。
- 任务能与角色、派系、区域和事件系统交叉。

## 11. Phase 7：NPC、遗物与旧世界内容

预计轮数：5-9 轮会话。

### 目标

- 设计重要 NPC 原型。
- 设计旧世界遗物与机器内容。
- 增强 CRPG 质感和探索动机。

### 内容目标

- 首批 30 个重要 NPC 原型。
- 首批 30-50 个遗物、旧机器、记录物。
- 每个重要 NPC 拥有恐惧、欲望、信仰偏差和关系钩子。

### 产出

- NPC 原型池。
- 记忆与关系字段。
- 遗物设计表。
- 旧世界碎片文本。
- NPC 事件参与规则。
- 遗物系统后果表。

### 验收标准

- NPC 能参与系统事件，而不是只提供对白。
- 遗物会改变信息、关系、资源或风险。
- 旧世界内容服务玩法和历史感。
- 每个重要 NPC 至少与一个派系、一个事件类型、一个任务目标关联。

## 12. Phase 8：首批高密度场景

预计轮数：5-8 轮会话。

### 目标

- 做 3 个 MVP 场景。
- 每个场景验证一种核心玩法。
- 每个场景支持不同角色的不同解法。

### 推荐场景

1. The Sealed Choir
   - 核心：信仰传播与隔离。
   - 验证：连接和切断如何改变文化扩散。

2. The Hungry Vein
   - 核心：资源流动与生存危机。
   - 验证：拓扑调整如何拯救或毁掉节点群。

3. The Cracking Crown
   - 核心：稳定性、派系利益、灾害管理。
   - 验证：坍塌风险如何成为政治问题。

### 产出

- 场景初始地图。
- 初始派系位置。
- 初始资源状态。
- 角色出生点。
- 场景目标。
- 事件链。
- 失败条件。
- 试玩验收标准。

### 验收标准

- 每个场景 10-20 分钟内能体验核心乐趣。
- 不同角色有明显不同路线。
- 场景能用于程序测试和策划试玩。
- 每个场景至少覆盖 3 个系统交叉点。

## 13. Phase 9：平衡、测试与内容扩展标准

预计轮数：持续进行；每个开发阶段后 1-2 轮。

### 目标

- 定义什么叫好玩。
- 定义什么叫无聊。
- 为 AI 生成和模拟验证提供评分规则。

### 评分维度

- 玩家决策有效性。
- 事件密度。
- 派系反应可读性。
- 危机节奏。
- 角色差异。
- 任务解法数量。
- 失败是否有意义。
- 世界历史是否可讲述。

### 产出

- Playability scoring rubric。
- Seed 评分表。
- 场景测试表。
- 内容扩展规范。
- 平衡调整记录。

### 验收标准

- 每个场景或 seed 能被量化评价。
- 能判断稳定但无聊的内容。
- 能指导下一轮内容补强。
- 评分结果能反馈给技术侧生成和验证系统。

## 14. 每轮统一自检标准

每一轮策划完成后，必须通过以下检查。

### 目标检查

- 本轮目标是否完成。
- 是否偏离当前 Phase。
- 是否产生可交付文档或内容表。

### 系统化检查

- 内容是否有字段。
- 是否有触发条件。
- 是否有数值范围或状态变化。
- 是否能被程序实现。

### 玩法检查

- 玩家是否能感知这个内容。
- 是否会产生选择。
- 是否有代价、风险或后果。
- 是否服务“拓扑改变文明”。

### 丰富度检查

- 是否只是概念，还是有足够样例。
- 是否能扩展成内容池。
- 是否和角色、派系、事件、任务产生交叉。

### 架构校对

- 是否符合已有 GDD / TDD / Project Plan。
- 是否破坏模拟优先原则。
- 是否要求程序做不必要的复杂架构。
- 是否能映射到数据模型。

### 反脆弱实现检查

- 是否出现重复字段。
- 是否有含糊概念无法落地。
- 是否把多个系统混成一团。
- 是否能拆成清晰模块。
- 是否避免临时特殊规则。

### 验证检查

- 是否能写出验收标准。
- 是否能被测试 AI 或程序 AI 检查。
- 是否有至少一个示例用例。
- 是否列出风险和未决问题。

## 15. 每轮通过后推送标准

每轮结束必须执行以下流程：

1. 更新或新增本轮文档。
2. 自检并修正。
3. 检查文档结构和命名。
4. 确认没有无关改动被纳入提交。
5. 提交 Git。
6. 推送远端。
7. 在总结中记录：
   - 完成内容。
   - 自检结果。
   - 推送 commit。
   - 下一轮目标。

推荐提交节奏：

- 每轮一个 commit。
- commit message 示例：
  - `Design phase A role system`
  - `Design faction content bible`
  - `Add event template library`
  - `Define scenario The Sealed Choir`

## 16. 推荐并行安排

程序员 Phase 1-2 时，策划做：

- Phase 0。
- Phase 1。
- Phase 2 第一版。

程序员 Phase 3 时，策划做：

- Phase 3。
- Phase 4。
- Phase 5 第一批事件。

程序员 Phase 4 时，策划做：

- Phase 2 完整派系。
- Phase 6。
- Phase 8 第一批场景。

程序员 Phase 5 以后，策划做：

- Phase 7。
- Phase 9。
- 内容扩展和平衡。

## 17. 结论

ROOTLINE 的策划目标可以总结为：

> 程序线负责让系统成立，策划线负责让系统值得玩、值得重玩、值得产生历史。

策划产出必须始终服务于可执行内容、可验证体验和可扩展内容池。
